# CS-250-FInal-Project

Project Summary: DriverPass is a driving school management system. The client wanted a web-based platform to help students prepare for their DMV driving tests. The system lets customers create accounts, buy lesson packages, book driving appointments, study materials, and take practice quizzes. It also supports different roles like drivers, a secretary, an IT officer, an owner, and the DMV, each with their own set of permissions and tasks.

What I did particularly well: The use case diagram came together really well. It clearly lays out who does what across all six actor roles, from the customer booking lessons to the owner pulling reports. The sequence diagram for the new appointment flow is another strong point. It walks through the entire process across nine components, including the login check, package hour validation, and the notification messages sent out to the driver and secretary at the end. That kind of detail shows a clear picture of how the system actually behaves behind the scenes.

What I would revise: I’d go back and rework the class diagram. The relationship between Customer and User feels a bit unclear. Customer is shown as its own class, but it really should inherit from User, and that connection isn’t fully fleshed out. I’d also add methods to each class instead of just listing attributes, and make sure every association has proper multiplicity labels (like showing that one customer can have many appointments, or one package can be tied to multiple customers). Those additions would make the diagram much more useful when it’s time to actually build the system.

Interpreting user needs: I took each stakeholder’s goals and turned them into specific system features. For example, the security requirement about locking accounts after 4 failed login attempts shows up directly in the login activity diagram. You can see the branching logic where the system counts attempts, displays errors, locks the account, and notifies the IT officer. The appointment scheduling activity diagram reflects the customer’s need for a smooth booking experience while still enforcing business rules, namely checking remaining package hours before confirming. Keeping user needs front and center matters because even the most technically solid system will flop if it doesn’t match how people actually want to use it. Design should reduce friction, not create it.

My approach to software design: I like to start broad and work my way down. That means gathering requirements first, then building out use case diagrams to capture what the system needs to do, followed by activity and sequence diagrams to figure out how it does it, and finally class diagrams to nail down the data structure. Going forward, I’d want to bring in more iterative prototyping, getting lightweight mockups in front of users early so I can catch misunderstandings before they get baked into the design. I’d also lean more on user stories to keep every design decision tied back to a real need, and look into domain-driven design to make sure the class structure actually mirrors how the business thinks about its own processes.
